#  整体架构设计

采用分层解耦的架构，避免 GUI、调度与硬件控制耦合在一起，便于后期扩展新仪器和新算法节点。

**UI 层**负责节点绘制、拖拽、连线交互；

**模型层**把图结构抽象为 NetworkX 的 `DiGraph`，作为 UI 与执行器之间的中间表示；

**执行引擎**负责拓扑排序与数据流动；

**驱动层**封装各类科研仪器；

**绘图层**专门处理高频数据的实时渲染。



### **GUI 与节点编辑器选型**

对于桌面端节点图交互，推荐以下几种成熟方案，按不同需求选择：

| 框架                     | 基础      | 优点                                                   | 适用场景                 |
| ------------------------ | --------- | ------------------------------------------------------ | ------------------------ |
| **NodeGraphQt**          | PySide2/6 | 开箱即用的节点图组件，支持拖拽、连线、右键菜单、序列化 | 首选，ComfyUI 风格最接近 |
| **Ryven**                | PySide    | 已是完整的 Python 可视化流式编程框架                   | 快速原型，可直接改造     |
| **PyFlow**               | Qt        | 节点式可视化脚本平台                                   | 需要脚本化扩展           |
| **Dear PyGui / imnodes** | DearPyGui | GPU 加速，性能极好                                     | 高刷新率、大量节点场景   |
| **Qt Quick + 自绘**      | QML       | 完全自定义视觉风格                                     | 追求独特 UI 设计         |

对于科研仪器场景，**NodeGraphQt** 是性价比最高的起点，它天然支持节点端口、连线校验、序列化为 JSON，配合 PySide6 可以直接嵌入你自己的主窗口、属性面板和绘图面板。

### **工作流模型与拓扑执行**

把节点图在内存中表示为 `networkx.DiGraph`，每个节点是一个计算单元（含输入端口、输出端口、参数、执行函数），每条边携带"源端口 → 目标端口"的映射。执行时的核心逻辑是：

1. 调用 `nx.topological_sort(G)` 得到执行顺序，确保无环（有环时提示用户）；
2. 按顺序执行每个节点的 `compute()`，把输出缓存在端口上；
3. 下游节点从其入边拉取上游输出作为输入；
4. 对"设备节点"或"长任务节点"使用 `asyncio` 或 `QThread`，避免阻塞 UI。

为了支持 ComfyUI 风格的**增量执行**（只重算参数变化的下游分支），可以给每个节点计算一个基于参数和上游输出的哈希值，哈希未变则跳过。ComfyUI 本身就是这种思路，你可以直接参考其 `execution.py` 的实现思路。

对于需要"流式"的仪器节点（如示波器持续采样），不适合按一次性 DAG 执行，建议引入**数据流模式**：节点启动后作为生产者在后台线程循环产出数据，通过 Qt 信号槽或 `queue.Queue` 推送给下游订阅节点。

### **设备控制层**

科研仪器的通信协议差异很大，建议做统一的驱动抽象基类 `InstrumentDriver`，再为每类总线派生实现：

- **GPIB / USB-TMC / LAN（SCPI 仪器）**：使用 [PyVISA](https://pyvisa.readthedocs.io/)，覆盖大多数示波器、频谱仪、电源、万用表。
- **串口设备**：使用 `pyserial`，适合自研 MCU、温控器、步进电机控制器。
- **NI 数据采集卡**：使用 `nidaqmx`，用于高速 AI/AO/DI/DO。
- **相机**：`pypylon`（Basler）、`PySpin`（FLIR）、`harvesters`（GenICam 通用）。
- **运动控制**：厂商 SDK 的 Python 封装，或通过 Modbus/EtherCAT。

每种仪器在图中表现为一个或多个节点，例如"连接"、"配置"、"触发采集"、"读取波形"。建议把**设备会话**（VISA resource、serial handle）作为单例资源管理器统一托管，避免节点重复打开同一端口。

### **流式绘图**

实时绘图是科研软件的核心诉求，推荐使用 [PyQtGraph](https://www.pyqtgraph.org/)，它基于 Qt + NumPy，专为高刷新率科学绘图设计，典型可以做到 60 FPS 下绘制百万点级曲线。实现要点：

- 使用**环形缓冲区**（`collections.deque(maxlen=N)` 或预分配的 NumPy 数组 + 写指针）保存最近 N 个样本；
- 采集线程把新数据 `emit` 给 UI 线程，UI 线程调用 `curve.setData()` 刷新；
- 刷新频率与采集频率解耦，例如采集 10 kHz、绘图 30 Hz，避免 UI 卡顿；
- 多通道用 `GraphicsLayoutWidget` 叠加多个子图；
- 需要频谱/瀑布图时可用 `ImageItem` + `scipy.signal.spectrogram`。

### **工作流保存、加载与导出**

把整张图序列化为 JSON 是最通用的方案，结构可以设计为：

```jso
{
  "version": "1.0",
  "nodes": [
    {"id": "n1", "type": "Oscilloscope", "pos": [120, 80], "params": {"channel": 1, "sample_rate": 1e6}},
    {"id": "n2", "type": "FFT", "pos": [360, 80], "params": {"window": "hann"}}
  ],
  "edges": [
    {"from": "n1.waveform", "to": "n2.signal"}
  ]
}

```

- **保存/加载**：NodeGraphQt 自带 `save_session` / `load_session`，也可自行实现；
- **版本兼容**：加入 `version` 字段，反序列化时做迁移；
- **导出为脚本**：遍历拓扑顺序，把每个节点翻译成等价的 Python 函数调用，生成可独立运行的 `.py` 文件，这对科研复现非常有价值；
- **导出数据**：为绘图节点加"导出 CSV / HDF5 / NPZ"按钮，大数据量优先用 `h5py`。

### **推荐的最小技术栈组合**

综合考虑开发效率、性能和社区生态，下面这套组合可以覆盖你列出的全部需求：

- **GUI 框架**：PySide6
- **节点编辑器**：NodeGraphQt
- **图模型与拓扑**：NetworkX
- **异步执行**：`asyncio` + `qasync`，或 `QThreadPool`
- **仪器通信**：PyVISA + pyserial + nidaqmx（按需）
- **数值计算**：NumPy + SciPy + Pandas
- **实时绘图**：PyQtGraph
- **数据持久化**：JSON（工作流）、HDF5（数据）
- **打包分发**：PyInstaller 或 Nuitka

### **落地实施建议**

第一阶段先打通"单节点 → 连线 → 拓扑执行 → 绘图"的最小闭环，用一两个虚拟仪器节点（例如正弦波发生器 + FFT + 曲线显示）验证框架。第二阶段接入真实仪器驱动，完善设备会话管理和错误处理。第三阶段补齐**增量执行、流式数据流、工作流导入导出、脚本导出**等高级特性。第四阶段再做插件机制，让用户可以通过放置 Python 文件自动注册新节点，这一点 ComfyUI 的 `custom_nodes` 目录是非常好的范本，可以直接借鉴其动态加载思路。

如果希望更快起步，也可以直接 fork [Ryven](https://ryven.org/) 或 ComfyUI 的前端架构，在其基础上替换节点库为科研仪器节点，这样能省去大量 UI 细节打磨的时间。
